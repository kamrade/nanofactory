# MVP-0 Current Functionality Spec

## 1. Общее описание

Продукт представляет собой SaaS-конструктор одностраничных сайтов в духе Carrd.

Ключевые принципы текущей версии:

- только одностраничные сайты;
- сайт собирается из предопределенного набора блоков;
- структура контента хранится как данные, а не как произвольный HTML;
- пользователь работает только со своими проектами;
- проект можно опубликовать и открыть по публичному URL;
- изображения загружаются отдельно и используются внутри блоков через asset references.

Текущая версия — это **MVP-0 functional baseline**: продукт уже покрывает полный базовый пользовательский цикл от логина до публикации.

- приложение задеплоено на dev-сервере;

## 4. Архитектурная основа

### Технологический стек

- Next.js
- Drizzle ORM
- PostgreSQL
- Auth.js
- Cloudflare R2 для файлов
- Docker / Dokploy для dev deployment

### Базовые архитектурные решения

- схема БД управляется из кода;
- `Project` является основной бизнес-сущностью;
- содержимое страницы хранится в `project_contents.content_json`;
- тема пока задается через `themeKey`;
- файлы хранятся в object storage;
- метаданные файлов хранятся в Postgres;
- блоки в контенте ссылаются на ассеты через `assetId`, а не через raw URL.

## 5. Доменная модель

## 5.1 User

Представляет владельца проектов.

Поддерживаемые поля:

- `id`
- `email`
- `displayName`
- `passwordHash`
- `createdAt`
- `updatedAt`

Назначение:

- аутентификация;
- владение проектами;
- привязка пользовательского контекста ко всем приватным действиям.

---

## 5.2 Project

Основная сущность продукта.

Поддерживаемые поля:

- `id`
- `userId`
- `name`
- `slug`
- `themeKey`
- `status`
- `publishedAt`
- `createdAt`
- `updatedAt`

Назначение:

- хранение метаданных лендинга;
- хранение статуса публикации;
- привязка к владельцу.

Поддерживаемые статусы:

- `draft`
- `published`

---

## 5.3 ProjectContent

Хранит редактируемое содержимое проекта.

Поддерживаемые поля:

- `id`
- `projectId`
- `contentJson`
- `schemaVersion`
- `createdAt`
- `updatedAt`

Назначение:

- хранение структуры страницы;
- хранение списка блоков и их props;
- источник данных для editor и public renderer.

### Формат контента

Контент хранится как JSON-структура вида:

```
typePageContent= {
  blocks:Array<{
    id:string;
    type:string;
    props:Record<string,unknown>;
  }>;
}
```

---

## 5.4 Asset

Представляет загруженный файл, используемый внутри проекта.

Поддерживаемые поля:

- `id`
- `projectId`
- `storageKey`
- `originalFilename`
- `mimeType`
- `sizeBytes`
- `width`
- `height`
- `alt`
- `createdAt`
- `updatedAt`

Назначение:

- хранение metadata о файле;
- связь файла с проектом;
- использование изображений в блоках.

Важное правило:

- физический файл хранится в Cloudflare R2;
- в Postgres хранятся только метаданные.

## 6. Аутентификация

Реализована аутентификация по email + password.

### Что поддерживается

- login через credentials flow;
- session-based доступ к приватной зоне;
<!-- REVIEW: Access control is implemented, but auth uses NextAuth with JWT session strategy, not a database-backed session strategy. -->
- logout;
- получение текущего пользователя на сервере;
- защита приватных маршрутов.

### Что используется

- Auth.js
- `passwordHash` в таблице `users`
- bcrypt-based проверка пароля

### Что пока не входит

- регистрация через UI;
- email verification;
- magic link;
- forgot password;
- OAuth providers.

### Текущий режим

Для разработки используется тестовый пользователь, созданный через seed.

## 7. Приватная зона приложения

Приватная часть приложения доступна только авторизованному пользователю.

### Что защищено

- dashboard;
- маршруты проекта / редактора;
- операции над проектами;
- операции над ассетами.

### Поведение

- неавторизованный пользователь редиректится на login;
- авторизованный пользователь может видеть только свои данные;
- чужие проекты и чужие ассеты недоступны.

---

## 8. Dashboard и проекты

### Реализовано

- просмотр списка своих проектов;
- создание проекта;
- открытие проекта;
- доступ к проекту только у владельца.

### При создании проекта

Автоматически создаются:

- запись в `projects`;
- запись в `project_contents` с пустым контентом.

Начальное содержимое:

```
{
  "blocks": []
}
```

### Ownership model

Все project-related операции выполняются от имени текущего пользователя, и на сервере проверяется принадлежность проекта через `userId`.

---

## 9. Editor

Реализован базовый редактор проекта.

### Что умеет editor

- загрузить проект и его контент;
- отобразить текущую структуру страницы;
- работать с блоками;
- сохранять изменения в БД.

### Поддерживаемые действия

- добавление блока;
- удаление блока;
- изменение props блока;
- сохранение обновленного `contentJson`.

### Block model

Editor работает через block registry / поддерживаемую модель блоков.

В рамках текущего состояния поддерживается базовый набор блоков, достаточный для построения первой версии лендинга.

### Валидация

На сервере выполняется базовая валидация структуры `contentJson`:

- `contentJson` должен быть объектом;
- `blocks` должен быть массивом;
- у блока должны присутствовать:
    - `id`
    - `type`
    - `props`

Также контролируются допустимые block types.

### Ownership

Сохранение editor content возможно только для владельца проекта.

---

## 10. Publish / Unpublish

Реализован базовый publish flow.

### Поддерживается

- публикация проекта;
- снятие проекта с публикации;
- изменение статуса проекта;
- установка/обновление publish-related полей проекта.

### Поведение

- публиковать и снимать с публикации может только владелец проекта;
- опубликованный проект становится доступен публично;
- неопубликованный проект не должен быть доступен через public route.

---

## 11. Public Rendering

Реализован публичный рендер опубликованного проекта.

### Публичный маршрут

- `/p/[slug]`

### Что делает public route

- ищет проект по `slug`;
- проверяет, что проект опубликован;
- загружает `project_contents`;
- рендерит страницу на основе сохраненного `contentJson`.

### Важные свойства

- публичный рендер использует тот же формат контента, что и editor;
- отдельного “формата публикации” нет;
- неопубликованный проект или неизвестный `slug` дают безопасное not found-поведение.

---

## 12. Assets: storage foundation

Поддержан базовый storage layer для работы с файлами.

### Выбранное решение

- Cloudflare R2

### Подготовлено

- server-side storage config;
- server-side storage client;
- helper генерации `storageKey`;
- helper построения public asset URL;
- storage service layer.

### Базовые принципы

- секреты доступны только серверу;
- storage concerns отделены от UI;
- URL-логика и storage key generation централизованы.

---

## 13. Assets: upload flow

Реализован backend flow загрузки изображений.

### Что поддерживается

- загрузка файла в контексте конкретного проекта;
- server-side ownership check;
- server-side validation файла;
- upload в Cloudflare R2;
- создание metadata в таблице `assets`.

### Поддерживаемые MIME types

- `image/jpeg`
- `image/png`
- `image/webp`

### Что пока не поддерживается

- SVG
- PDF
- видео
- произвольные бинарные файлы
- image editing / crop pipeline

### Ограничения

- максимальный размер файла ограничен;
- валидация идёт на сервере;
- при невалидном файле запись в БД не создаётся.

---

## 14. Assets: editor integration

Реализована базовая интеграция ассетов в редактор.

### Что поддерживается

- список ассетов проекта;
- выбор ассета для поддерживаемого блока;
- сохранение `assetId` в `contentJson`;
- отображение превью в editor.

### Принцип хранения

В блоке хранится не URL, а ссылка на ассет:

```
{
  "type":"hero",
  "props": {
    "imageAssetId":"..."
  }
}
```

### Безопасность

При сохранении и использовании ассета проверяется, что:

- ассет существует;
- ассет принадлежит тому же проекту;
- проект принадлежит текущему пользователю.

---

## 15. Assets: public rendering

Реализован публичный рендер изображений через `assetId`.

### Что происходит

- public renderer читает `assetId` из блока;
- выполняется resolution ассета;
- `storageKey` преобразуется в public URL;
- изображение отображается на публичной странице.

### Важные свойства

- чужие ассеты не должны подтягиваться в проект;
- невалидный или отсутствующий ассет не должен ломать страницу;
- content format не меняется ради public rendering.

---

## 16. Ownership и безопасность

Во всей системе соблюдается единая модель владения данными.

### Основные правила

- пользователь видит только свои проекты;
- пользователь работает только со своими проектами;
- пользователь загружает ассеты только в свои проекты;
- пользователь использует только ассеты своих проектов;
- публичный рендер не раскрывает неопубликованные проекты.

### Серверные проверки выполняются для:

- login/session-based доступа;
- чтения проекта;
- сохранения проекта;
- публикации;
- загрузки файла;
- получения списка ассетов;
- использования `assetId`.

## 19. Что сознательно не входит в текущий MVP-0

Следующие вещи **не входят в текущий функциональный baseline**:

- editor UX polish как отдельный этап;
- полноценная theme system;
- registration page;
- email verification;
- magic link login;
- custom domains;
- subdomain publishing;
- version history;
- multi-page сайты;
- advanced media management;
- asset editor / cropper;
- forms / lead capture;
- analytics;
- billing;
- collaboration / teams.

## 20. Текущий end-to-end user flow

На текущем этапе поддерживается следующий базовый сценарий:

1. Пользователь логинится.
2. Пользователь попадает в dashboard.
3. Пользователь создаёт проект.
4. Пользователь открывает editor.
5. Пользователь добавляет и редактирует блоки.
6. Пользователь загружает изображение.
7. Пользователь привязывает изображение к поддерживаемому блоку.
8. Пользователь сохраняет проект.
9. Пользователь публикует проект.
10. Проект открывается по публичному URL.
11. Публичная страница рендерится вместе с изображениями.
