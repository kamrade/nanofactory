ТЗ: Шаг 8 — Assets MVP
Цель

Добавить первую рабочую поддержку ассетов в продукт.

После этого шага пользователь должен иметь возможность:

загрузить изображение в рамках проекта
сохранить metadata ассета в БД
выбрать загруженный ассет в editor
использовать assetId внутри block props
увидеть изображение в editor и на публичной странице

На этом шаге нам не нужны продвинутые media features.
Нужен рабочий MVP-флоу.

Что нужно сделать
1. Подключить Cloudflare R2

Нужно подготовить storage layer для загрузки файлов в Cloudflare R2.

Требования
использовать R2 как object storage
хранить физические файлы в R2
не хранить binary data в Postgres
подготовить env-переменные для R2

Минимально нужны:

account / endpoint config
bucket name
access key
secret key
public asset base URL
2. Сделать server-side storage client

Нужен отдельный серверный модуль для работы с R2.

Он должен уметь минимум:

загружать файл по storageKey
при необходимости формировать public URL
Требования
не размазывать storage-логику по route/page components
не вызывать R2 напрямую из клиентского UI
3. Реализовать upload endpoint / server action

Нужна серверная точка загрузки файла.

Допустимо:

route handler
server action
другой принятый в проекте server-side способ
Флоу
пользователь выбирает файл в editor
сервер валидирует файл
сервер генерирует storageKey
файл загружается в R2
metadata сохраняется в таблицу assets
клиент получает созданный asset record или нужный минимум данных
4. Валидировать загружаемые файлы

На этом шаге поддерживаем только безопасный минимальный набор.

Разрешённые MIME types
image/jpeg
image/png
image/webp
Не поддерживать пока
image/svg+xml
pdf
видео
любые произвольные файлы
Ограничения
max file size: до 10 MB
валидировать MIME type
желательно валидировать extension
при возможности извлечь width и height

Если файл невалидный:

вернуть понятную ошибку
не загружать его в storage
не создавать запись в БД
5. Сохранять metadata в assets

После успешной загрузки в R2 нужно создавать запись в таблице assets.

Минимально заполнять:

projectId
storageKey
originalFilename
mimeType
sizeBytes
width
height
alt (можно пустым / nullable)
6. Ownership и безопасность

Ассеты в MVP являются project-owned.

Требования
загружать ассеты может только владелец проекта
нельзя загружать файл “в чужой проект”
нельзя получить список/выбрать ассеты чужого проекта
все server-side queries должны учитывать ownership
7. Сделать project asset queries

Нужны server-side queries минимум для:

createAssetForProject(...)

Создание metadata после успешного upload

getAssetsByProjectIdForUser(projectId, userId)

Список ассетов проекта только для владельца

getAssetByIdForProject(...) или аналогичный helper

Для безопасного выбора/разрешения ассета

8. Добавить asset picker в editor

В editor нужно добавить минимальный UI для выбора изображения.

На этом шаге не нужен полноценный media manager.

Минимально достаточно:

загрузить файл
увидеть список уже загруженных ассетов проекта
выбрать ассет для поддерживаемого блока
9. Интегрировать assets в block props

Нужно использовать assetId внутри block props, а не raw URL.

Пример:

{
  "id": "hero-1",
  "type": "hero",
  "props": {
    "title": "Hello",
    "imageAssetId": "..."
  }
}
Требования
editor сохраняет assetId
renderer не хранит и не ожидает raw R2 URL внутри contentJson
10. Поддержать минимум один image-aware block

На этом шаге достаточно интегрировать assets хотя бы в один блок.

Рекомендуемый вариант:

hero

Например:

title
subtitle
buttonText
imageAssetId

Если уже есть другой блок, куда картинка ложится естественнее, можно использовать его.

11. Реализовать asset URL resolution

Нужен понятный server-side способ резолвить:

assetId
или asset metadata

в публичный URL изображения.

Требования
renderer должен получать URL через storage layer / resolver
не хардкодить raw storage URL по месту
подготовить почву для будущих transformations или смены storage provider
12. Отобразить изображения в editor и public renderer

После выбора ассета:

editor должен уметь показать превью изображения
public page должна рендерить это изображение
Что не нужно делать на этом шаге

Не нужно:

делать полноценную media library между проектами
делать folders / tags
делать delete/replace asset flow, если это сильно тормозит шаг
делать image transformations
делать thumbnails pipeline
делать background jobs
делать quotas / billing integration
делать drag-and-drop upload polish
поддерживать SVG
Ожидаемый результат

После выполнения шага должно быть так:

Владелец проекта может
загрузить изображение в рамках проекта
увидеть загруженный ассет
выбрать его в поддерживаемом блоке
сохранить assetId в content
увидеть картинку в editor
увидеть картинку на публичной странице после publish
В системе
файл физически лежит в R2
metadata лежит в Postgres
contentJson хранит assetId, а не сырой URL
Требования к качеству реализации
По архитектуре
storage layer отделён от UI
upload flow server-side
DB metadata и physical storage разделены
asset resolution не размазана по компонентам
По безопасности
ownership checks server-side
валидация файла server-side
нельзя загружать и использовать ассеты чужого проекта
По DX
структура должна готовить почву для:
image blocks beyond MVP
future asset deletion
future transformations
future media library
Критерий готовности

Шаг считается завершённым, если:

Cloudflare R2 подключён
есть server-side upload flow
файл валидируется и загружается в R2
metadata сохраняется в assets
editor умеет выбирать ассет минимум для одного блока
assetId сохраняется в contentJson
изображение отображается в editor и на публичной странице
ownership checks соблюдены
агент явно указал:
где находится storage client
как устроен upload flow
какие MIME types поддерживаются
в какой блок интегрированы assets
как резолвится public asset URL
Что останется после Шага 8

После этого останется 2 основных продуктовых шага.

Шаг 9 — Editor UX upgrade

Сделать:

reorder blocks
duplicate block

более удобные панели редактирования
Шаг 10 — Theme system proper

Сделать:

theme registry

единый визуальный язык editor/public rendering
